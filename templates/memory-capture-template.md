# Memory 沉淀模板

> 只记录长期有效、可复用的结论；不要直接记录一次性聊天内容。
> 按四分类归档，每条经验归入最匹配的一类。

## 分类

选择一个：
- **Pattern** — 可复用的模式和做法（如"分页查询的参数校验逻辑"）
- **Pitfall** — 踩过的坑和需要避免的做法（如"下游 Svrkit 超时需降级处理"）
- **Preference** — 命名偏好、约定（如"查询接口统一用 GET /api/{resource-name}"）
- **Architecture** — 架构决策和技术选型（如"轻量逻辑放 entity/，不加 service 层"）

## 排除原则（什么不该记）

> 来源：voucher/mkt 团队 OpenSpec 知识文档提炼策略。

以下内容**不应**写入 Memory：
- **代码能直接表达的流程逻辑**——通过读代码即可理解的控制流、调用链、分支判断。流程逻辑以代码为准，写进文档会导致代码和文档双份维护、口径不一致。
- **一次性聊天内容**——临时讨论、调试过程、单次问答。
- **证据不足的推测**——没有代码或运行结果佐证的猜测。

只文档化满足以下**全部条件**的知识：
1. **AI 读代码后仍不理解的**——隐性业务规则、跨系统约定、历史遗留原因
2. **业务强相关的**——领域概念、状态机、金额守恒等业务约束
3. **相对稳定的**——短期内不会频繁变化的规则和约定
