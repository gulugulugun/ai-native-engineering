# 需求与分析阶段

你正在执行 Implement 流程的**第 1 阶段：需求与分析**。

## 执行步骤

1. 从需求描述中提取功能点和业务规则。
2. 对每条规则做四要素结构化：
   - 规则描述
   - 满足时行为
   - 违反时行为
   - 边界条件
3. 主动挑战前提假设（Premise Challenge）：
   - 这是对的问题吗？
   - 如果什么都不做会怎样？
   - 现有代码中有什么可复用的？
4. **代码库搜证**：对分析过程中产生的待确认问题，先在代码库中搜索答案：
   - 在旧前端/旧后端代码中搜相关接口、常量、枚举定义
   - 在新模板中搜已有的类似实现
   - 找到证据 → 标记为"已从代码确认"并附带代码引用（文件路径 + 行号）
   - 找不到 → 标记为"待人工确认"
5. 产出 `requirement-analysis.md`，包含：
   - 功能点清单
   - 业务规则（四要素格式）
   - 风险初评
   - 待确认问题（标注"已从代码确认"或"待人工确认"）
   - **需求就绪度评估**（见下）

## 需求就绪度评估

在 `requirement-analysis.md` 末尾输出就绪度评级：

```markdown
## 需求就绪度

| 维度 | 状态 | 说明 |
|------|------|------|
| 业务目标 | ✅/⚠️/❌ | ... |
| 功能点完整性 | ✅/⚠️/❌ | ... |
| 业务规则 | ✅/⚠️/❌ | ... |
| 异常流程 | ✅/⚠️/❌ | ... |
| 待确认问题 | ✅/⚠️/❌ | X 个已确认 / Y 个待人工确认 |

**就绪度：HIGH / MEDIUM / LOW**
```

- **HIGH**：所有维度 ✅，可直接进入 spec
- **MEDIUM**：有 ⚠️ 但无阻塞性问题，建议先解决再进入 spec
- **LOW**：有 ❌ 或阻塞性问题未解决，**不建议进入 spec**

## 门禁

产出 `requirement-analysis.md` 后，**必须请人确认分析结果**，确认后才能进入下一阶段（spec）。

## 注意

- 不要在这个阶段写代码。
- 不要跳过前提挑战直接进入功能点提取。
- 规则穷举策略：CRUD 补全 / 正反面 / 边界值 / 异常流 / 交叉影响。
