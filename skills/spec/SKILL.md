# 规范定义阶段（Spec）

你正在执行 Implement 流程的**第 2 阶段：规范定义**。

本阶段回答"做什么"，下一阶段（design）回答"怎么做"。

## 前置条件

`requirement-analysis.md` 已完成且人已确认。

## 场景判断（先做）

基于 `requirement-analysis.md`，先判断需求场景类型：

| 场景 | 判断标准 | spec 侧重 |
|------|---------|-----------|
| **纯前端** | 仅涉及 UI 新增/调整，无新后端接口 | §2 侧重页面路由、组件层次、数据流；§3 只涉及前端文件 |
| **纯后端** | 仅涉及接口/逻辑变更，无前端页面 | §2 侧重 API 定义、错误码；§3 只涉及后端文件 |
| **全栈** | 前后端都需要变更 | §2 同时定义 API 和页面路由；§3 分前端和后端两组文件 |

> 纯前端场景可调用 `ui-guide` Skill 获取布局模板参考。

## 执行步骤

1. 基于 `requirement-analysis.md`，编写 `spec.md`，包含以下必选章节：

   **§1 目标（Goals）**
   - 一句话描述本需求要实现什么
   - 成功标准：怎样算"做完了"

   **§2 接口定义（API / Interface）**
   - 新增或修改的接口：路径、方法、请求参数、响应结构、错误码
   - 如果是前端需求：页面路由、组件层次、数据流

   **§3 文件规划（File Plan）**
   - 将创建的新文件及其职责
   - 将修改的已有文件及修改点
   - 遵循项目已有的目录结构和命名约定

   **§4 测试策略（Test Strategy）**
   - 基于 `requirement-analysis.md` 中的业务规则，列出需要覆盖的测试维度
   - 正向 / 反向 / 边界 / 权限

   **§5 约束与边界（Constraints）**
   - 技术约束（框架限制、平台限制）
   - 业务约束（权限、数据范围、性能要求）
   - 明确标注"不做什么"（Out of Scope）

2. 产出 `spec.md`。

## 门禁

产出 `spec.md` 后，**必须请人审查确认**，确认后才能进入下一阶段（design）。

## 注意

- 本阶段只定义"做什么"，不拆任务、不写代码。
- spec 是后续所有阶段（design、coding、testing、review）的**唯一锚点**。
- 禁止使用"TBD""待定""后续补充"等占位符——spec 里的每一项都必须是确定的。
