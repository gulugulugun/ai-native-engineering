# 测试与验证阶段

你正在执行 Implement 流程的**第 5 阶段：测试与验证**。

## 前置条件

编码阶段所有任务已完成且验证通过。

## 执行步骤

1. 基于 `requirement-analysis.md` 中的业务规则生成测试用例：
   - 满足时 → 正向用例
   - 违反时 → 反向用例 + 错误码验证
   - 边界条件 → 边界值测试
2. 产出 `test-cases.md`（结构化测试用例清单）。
3. 执行验证：
   - lint 检查
   - 编译检查
   - 契约校验（`openapi.yaml` 与实际接口是否一致）
4. 每次声称测试通过必须附带命令输出证据（退出码 + 通过/失败数）。

## 验证失败回流机制

验证发现问题时，**不要默认"改代码再跑"**。先判断问题出在哪一层，再决定回到哪里修：

| 根因分类 | 判断标准 | 回流目标 |
|---------|---------|---------|
| 🐛 实现问题 | 代码 bug、逻辑错误、功能遗漏 | 当前阶段直接修复 |
| 📋 任务拆解问题 | 任务粒度不对、依赖关系错误、DoD 不清晰 | → 回到 design 阶段调整 `tasks.md` |
| 📐 规格问题 | spec 有歧义、接口定义缺失、约束遗漏 | → 回到 spec 阶段修订 `spec.md` |
| 🔎 需求问题 | 需求理解错误、场景遗漏、业务规则有误 | → 回到 requirement-analysis |

**关键原则**：如果一个问题跨层，**优先归到更上层**——上层修复能从根本解决，避免在代码层反复打补丁。

回流时需要说明：
1. 回到哪个阶段
2. 为什么不是代码层能修的
3. 回流后需要重点关注什么

## 门禁

**必须请人验收最终结果**，确认关键场景覆盖后才能进入下一阶段（archiving）。

## 注意

- 不接受"应该能过""之前跑过了"等推断。
- 测试用例应覆盖所有四要素结构化规则，不遗漏。
- 验证失败时先做根因分类，不要直接改代码。
