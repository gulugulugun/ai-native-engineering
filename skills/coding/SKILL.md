# 编码实现阶段

你正在执行 Implement 流程的**第 4 阶段：编码实现**。

## 前置条件

`spec.md` 和 `tasks.md` 已完成且人已确认。

## 执行步骤

### 0. 编码前契约一致性校验（全栈/后端场景）

如果 `spec.md` §2 定义了后端 API 接口：

1. 检查 `{BE_OPENAPI_FILE}`（`.xdc/contract/openapi.yaml`）中是否已有对应接口定义
2. 如果本地契约与 `spec.md` 的接口定义不一致，先更新契约
3. 使用 XContract MCP 查询远程契约，确认本地和远程一致
4. 如有差异，提醒人确认：是更新本地还是以本地为准

> 契约一致性校验通过后，再进入编码。

### 1. 逐任务编码

1. 先更新契约（`openapi.yaml`），再写实现代码。
2. 按 `tasks.md` 逐任务执行，不跳步。
3. 每个任务完成后：
   - 运行验证命令（lint / 编译 / 契约校验）
   - 读取完整输出
   - 确认通过后才声称完成
4. 遵循项目 Rules：
   - 前端：Pages↔Models↔Services 三层分离，路由注册在 `src/index.ts`
   - 后端：Controller 以契约生成结构为准，业务逻辑放 `entity/`，异常用 `BizError`

## 禁止事项

- 禁止"应该能过""大概完成了"等无证据声称。
- 禁止使用"TBD""TODO"等占位符。
- 禁止"与任务 N 类似"等模糊引用，必须展开代码。
- 禁止在测试之前声称功能完成。
- 禁止未经要求的额外重构、额外注释、额外类型标注。bug 修复不需要顺便清理周边代码，简单功能不需要额外的可配置性。
- 禁止过早抽象：三行相似代码优于一个只用一次的 helper 函数。不要为假想的未来需求设计。
- 禁止无用的兼容性保留：未使用的代码直接删除，不要重命名为 `_var`、不要 re-export、不要加 `// removed` 注释。
- 禁止过度防御编程：只在系统边界做输入验证（用户输入、外部 API），内部代码信任框架保证，不要为不可能发生的场景加错误处理。
- 禁止修改未读的文件：要改一个文件，必须先读它。
- 禁止创建不必要的新文件：优先编辑已有文件，除非确实需要新建。

## 完成条件

所有任务完成且验证通过后，进入编码审查：

1. **自查**：lint 通过 + 契约校验通过 + 无 TODO/TBD 残留。
2. **人审查关键逻辑**：请人确认核心业务逻辑、权限校验、下游调用的正确性。
3. 审查通过后，进入下一阶段（testing）。
