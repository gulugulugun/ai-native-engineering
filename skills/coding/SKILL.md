# 编码实现阶段

你正在执行 Implement 流程的**第 4 阶段：编码实现**。

本阶段不再“一口气做完全部任务”，而是按 `tasks.md` **逐批次实现 → 每批结束停下 → 等用户回复“继续”**。

## 前置条件

- `spec.md` 和 `tasks.md` 已完成且人已确认。
- `tasks.md` 必须使用批次化结构；若仍是旧版逐文件散列任务清单，应先回到 design 阶段重构。

## 核心升级

- `tasks.md` 同时充当批次计划与状态文件，状态以 `- [ ] / - [x]` 为准。
- 编码优先交给子代理 `batch-implementer`（或等价的 `Task / Subagent` 调用）执行，主流程只负责读取结果、展示证据和控制门禁。
- 每个批次结束后都必须 **STOP**，由人审查并决定是否继续。

## 执行步骤

### 0. 编码前契约一致性校验（全栈 / 后端场景）

如果 `spec.md` §2 定义了后端 API 接口：

1. 检查 `{BE_OPENAPI_FILE}`（`.xdc/contract/openapi.yaml`）中是否已有对应接口定义。
2. 如果本地契约与 `spec.md` 的接口定义不一致，先更新契约。
3. 使用 XContract MCP 查询远程契约，确认本地和远程一致。
4. 如有差异，提醒人确认：是更新本地还是以本地为准。

> 契约一致性校验通过后，再进入当前批次编码。

### 1. 锁定当前批次

1. 读取 `tasks.md`。
2. 找到按顺序排列的第一个包含 `- [ ]` 任务项的批次。
3. 如果不存在未完成批次，进入“完成条件”。
4. 不允许跳过前面的未完成批次，也不允许预先实现后续批次的代码。

### 2. 调用单批次编码子代理

1. 优先调用子代理 `batch-implementer`；如果运行环境不支持显式点名子代理，则用 CodeBuddy 原生 `Task / Subagent` 能力按同等职责隔离执行。
2. 子代理只能做三件事：
   - 实现**当前批次**涉及的代码
   - 运行当前批次要求的验证
   - 在验证通过后，把当前批次对应任务从 `- [ ]` 更新为 `- [x]`
3. 子代理禁止：
   - 跨批次实现代码
   - 擅自重写 `tasks.md` 的批次结构
   - 未验证就把任务标记为完成

### 3. 读取结果并出示证据

子代理返回后，主流程必须读取并展示：

- 当前批次编号与名称
- 已完成任务
- 修改文件
- 验证命令与通过 / 失败结果
- 是否存在阻塞项
- 下一批次编号与名称

如遇问题，先做根因分类：

- **实现问题**：在当前批次内修复
- **任务拆解问题**：回到 design 阶段调整 `tasks.md`
- **规格问题**：回到 spec 阶段修订 `spec.md`

### 4. 批次门禁

每个批次结束后，输出批次报告并 **STOP**：

- 用户回复 **“继续”** → 进入下一批次
- 用户提出修改意见 → 留在当前批次继续修正
- 若当前批次仍有未完成任务或验证失败 → 禁止推进下一批次

建议批次报告格式：

```text
✓ 批次 N「批次名称」已完成
- 完成任务：...
- 修改文件：...
- 验证结果：...
- 下一批次：批次 M「批次名称」/ 无
请审查代码，回复“继续”执行下一批次，或直接提出修改意见。
```

## 禁止事项

- 禁止“应该能过”“大概完成了”等无证据声称。
- 禁止使用 `TBD`、`TODO` 等占位符。
- 禁止“与任务 N 类似”等模糊引用，必须展开代码。
- 禁止在测试之前声称功能完成。
- 禁止未经要求的额外重构、额外注释、额外类型标注。
- 禁止因为“顺手”提前推进下一批次。
- 禁止修改未读的文件；要改一个文件，必须先读它。
- 禁止创建不必要的新文件：优先编辑已有文件，除非确实需要新建。

## 完成条件

所有批次完成且验证通过后，进入编码审查：

1. **自查**：lint 通过 + 契约校验通过 + `tasks.md` 中无遗漏的 `- [ ]` 任务。
2. **人审查关键逻辑**：请人确认核心业务逻辑、权限校验、下游调用的正确性。
3. 审查通过后，进入下一阶段（testing）。
