# 能力 Spec 主线目录

> 本目录存放**已部署能力的持久化 spec**。每个文件代表一个已经实现并验证过的能力，是该能力的"当前真相"。

## 什么是 Spec 主线

归档阶段完成后，当次需求的 `spec.md` 中涉及的能力描述会被**合并**到本目录下对应的主线文件中。这样 AI 在后续需求中能看到已部署能力的演进视图，而不是每次从零理解。

灵感来源：voucher/mkt 团队 OpenSpec 实践中的 `openspec/specs/<capability>/spec.md` 活文档机制。

## 命名规则

- 文件名：`<capability-name>.md`，kebab-case，动词-名词结构
- 示例：`open-order-query.md`、`deposit-detail-view.md`
- 一个文件对应一个独立能力，跨前后端的能力用一个文件（能力视图是跨项目的）

## 文件结构

```markdown
# 能力名称

## 概述
一两句话说明这个能力做什么。

## 当前状态
- 最后更新：YYYY-MM-DD
- 来源需求：对应的 spec.md 或需求描述

## 关键行为
### 行为 1：行为名称
系统 SHALL ...
### 行为 2：行为名称
系统 SHALL ...

## 涉及文件
- 前端：`src/pages/...`
- 后端：`src/controller/...`

## 变更历史
| 日期 | 变更内容 | 来源 |
|------|---------|------|
| YYYY-MM-DD | 初始创建 | spec.md |
```

## 何时更新

- **归档阶段（archiving Skill 步骤 5）**：当次 spec 中的能力描述合并到对应主线文件
- **新文件**：如果本目录下没有对应能力的文件，新建一个
- **已有文件**：在"关键行为"和"变更历史"中追加本次变更，不重写已有内容

## 何时读取

- **需求分析阶段**：了解已有能力，避免重复建设
- **spec 阶段**：对齐已有行为定义，确认是新增还是修改
- **design 阶段**：确认现有实现范围，辅助任务拆解
