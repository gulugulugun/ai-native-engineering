# AI 原生研发闭环体系方案（推荐命名）

> 版本：v0.1  
> 更新日期：2026-04-03  
> 适用范围：当前工作区四个项目（旧前端 `payclient-oa-depositmisview`、旧后端 `lqp`、新前端模板 `queryopenorderlist`、新后端模板 `xdc_wxpay`）

## 1. 命名建议

### 推荐名称
**AI 原生研发闭环体系**

### 推荐理由
- **不是单点的“AI 驱动写代码”**：目标覆盖需求分析、方案设计、契约定义、代码生成、测试验证、交接沉淀。
- **不是泛化的“以 AI 为基础”**：这个说法偏静态，不能体现流程闭环与工程体系。
- **“AI 原生”更贴近行业语境**：强调系统从一开始就把 AI 作为一等能力接入研发流程，而不是事后外挂一个助手。
- **“闭环体系”更准确**：突出从输入需求到输出可验证结果，并沉淀经验反哺后续任务。

### 可选名称
- `AI 原生研发体系`
- `AI 辅助研发闭环`
- `AI 原生工程执行体系`

> 当前建议：正式方案文档中统一使用 **AI 原生研发闭环体系**。

---

## 2. 方案目标

构建一套面向当前业务项目的 **AI 原生研发闭环体系**，让 AI 不只负责生成代码，而是能够在较少人工干预下，参与并推进以下完整流程：

1. 需求理解与约束识别
2. 方案设计与任务拆解
3. 契约优先的接口/数据定义
4. 前后端代码生成与修改
5. 测试、校验与验证
6. 交付后的经验沉淀与规则更新

### 一句话目标
**通过 Rules + Hooks + Memory + MCP + 模板化 SOP，把 AI 从“代码补全工具”提升为“可治理的研发执行体”。**

---

## 3. 设计背景：为什么现在要做这件事

当前团队的目标不是做一个通用 AI 产品，而是希望在现有业务项目中，建立一套 **可以持续复用、可校验、可沉淀** 的 AI 研发方式，用于加速：

- 新需求开发
- 旧业务迁移重构
- 前后端契约协同
- 测试验证与交付
- 跨会话 handoff 与团队协作

现状上，工作区已经具备：
- 新前端模板：`queryopenorderlist`（Vue 3 + Vite + TDesign + XPage）
- 新后端模板：`xdc_wxpay`（XDC Node SDK + OpenAPI 契约）
- 旧系统作为迁移参考：`payclient-oa-depositmisview`、`lqp`

因此，这套体系的定位是：**围绕当前四个项目，建立项目内可落地的 AI 研发基础设施**。

## 4. 方案设计参考来源与对应关系

本方案不是凭空拍脑袋设计，而是刻意对齐了业内较有共识、较权威的现代实践与官方能力。

### 4.1 Harness Engineering 思想参考

#### 参考思想
OpenAI 提出的 Harness Engineering 强调：

> 模型负责智能，Harness 负责提供手、眼、记忆和安全边界。

核心不是“让模型更会写代码”，而是“让模型在一个可靠的工程环境里工作”。

#### 对应到本方案
- **Rules**：提供项目约束与背景知识
- **Hooks**：提供运行时治理、拦截和验证
- **Memory**：提供跨会话长期记忆
- **模板 / SOP**：提供稳定执行路径
- **MCP / 工具**：提供外部系统能力接入

#### 对应结论
本方案借鉴了 `Harness Engineering` 的核心思想，但正式定位并不是构建一个通用 `Harness`，而是优先基于 `CodeBuddy` 原生能力，在当前工作区中建设面向业务研发场景的 **AI 原生研发闭环体系**。

### 4.2 OpenHarness 开源仓库

#### 参考实践
从 `OpenHarness` 仓库中重点学习到这些模式：
- 分层上下文装配
- Skill / SOP 按需加载
- 持久化记忆
- Hook 运行时治理
- 模式化工作流
- 会话交接与恢复

#### 对应到本方案
- **Rules 分层**：对应分层上下文装配
- **SOP 模板**：对应 Skill/SOP 机制
- **MEMORY + 模板**：对应持久化记忆与 handoff
- **Hooks MVP**：对应运行时治理
- **后续模式化工作流**：对应 plan/default/auto 一类工作模式

#### 对应结论
我们学习 `OpenHarness` 的重点，不是复制一个新框架，而是提炼其 **可迁移的工程模式**，再映射到 `CodeBuddy` 原生能力上。

### 4.3 CodeBuddy 官方能力

#### 参考文档
- `CodeBuddy Hooks` 官方文档
- `CodeBuddy Rules` / IDE 原生工作流能力

#### 已确认能力
- 支持项目级 `Rules`
- 支持 `SessionStart` / `PreToolUse` / `PostToolUse` 等 Hook 事件
- 支持 `allow / deny / ask` 类型的治理决策
- 支持工具调用与命令执行的运行时拦截

#### 对应结论
这意味着我们当前阶段 **没有必要先引入额外框架**，而是应先把 `CodeBuddy` 原生能力用足，做出最小可用版本。

### 4.4 契约优先（Contract-First）

#### 参考来源
结合当前 XDC / XContract 的现有机制，已经明确：
- 接口定义可以先通过契约（proto / yaml / OpenAPI）表达
- 代码骨架可由工具生成
- 前后端可以围绕同一份契约协作

#### 对应到本方案
- AI 的第一步不是直接写 Controller 或页面接口调用，而是：
  1. 查现有契约
  2. 修改或新增契约
  3. 校验契约
  4. 再生成和补业务代码

#### 对应结论
**契约驱动** 是这套体系能否稳定落地的关键支点，因为它把“需求 → 接口 → 代码”之间的关系结构化了。

## 5. 核心设计原则

### 5.1 契约驱动，接口定义先行
- 后端开发先定义契约，而不是先写实现
- 前后端都围绕契约协同
- AI 优先处理结构化定义与生成流程

### 5.2 代码第一性，Rules 只补盲区
- 代码和 MCP 能实时得到的信息，不重复堆进 Rules
- Rules 只记录“代码看不出来但 AI 必须知道”的东西
- 例如平台约束、历史坑点、宿主行为、禁忌做法

### 5.3 反馈闭环，做完即沉淀
- 每次真实需求结束后，都要更新 Rules / MEMORY / 模板
- 目标不是一次写完一套完美规则，而是在真实场景里不断收敛

### 5.4 原生能力先行，框架后置评估
- 先用 `CodeBuddy` 原生能力做最小可用体系
- 只有当原生能力不足以支撑治理链、插件生态、多 Agent 等场景时，再考虑框架层

## 6. 体系结构设计

```text
需求输入
  ↓
Rules（项目规则 / 架构规则 / 工作流规则）
  ↓
任务上下文（current-task / issue / PR 摘要）
  ↓
AI 执行（分析 / 设计 / 改代码 / 调 MCP / 调工具）
  ↓
Hooks 治理（SessionStart / PreToolUse / PostToolUse）
  ↓
测试与验证（lint / test / 契约校验 / 人工确认）
  ↓
Handoff / MEMORY 沉淀
  ↓
反哺下一轮任务
```

### 关键组成

#### A. Rules 层
作用：提供稳定背景与开发约束。

当前已落地：
- `workspace-architecture`
- `xpage-frontend-guardrails`
- `xdc-backend-contract-guardrails`
- `delivery-workflow`

#### B. Hooks 层
作用：提供运行时治理与风险控制。

当前已落地：
- `SessionStart`：注入当前项目关键上下文
- `PreToolUse`：拦截高危命令、敏感路径写入与删除

#### C. Memory / Context / Template 层
作用：区分长期知识、临时上下文和交付模板。

当前已落地：
- `MEMORY.md`
- `context/current-task.md`
- `templates/handoff-template.md`
- `templates/memory-capture-template.md`

#### D. MCP / 外部能力层
作用：把契约管理、代码仓库、MR 流程、平台能力等外部系统接进 AI 工作流。

当前重点：
- `xcontract` 相关 MCP 工具
- `工蜂` / Git 相关能力（代码提交、分支推送、MR 创建与查询）
- 后续可扩展到 issue/PR 同步、测试能力等

##### 关于工蜂的定位
- **工蜂 MCP** 更适合承接远端仓库与 MR 相关操作，如仓库信息查询、分支/MR 查询、MR 创建或更新。
- **本地代码提交与 push** 仍更适合通过本地 Git CLI 完成；工蜂更适合作为远端协作与 MR 自动化能力。
- 工蜂支持通过 `git push -o mr.create` 等 `push options` 在推送代码时直接创建 MR，并可附带标题、描述、评审人等参数。
- 如果后续接入完整版工蜂 MCP，可进一步把“代码修改完成 → 提交分支 → 创建/更新 MR → 查询评审状态”串成更完整的 AI 交付链路。
- 需要注意：工蜂 MCP 偏向远端仓库 API，不承担本地 clone 能力，因此本地仓库工作流仍需与 Git CLI 配合。

## 7. 当前阶段的实施策略

### 第一阶段：最小可用原型（已完成）
- 建立第一版 Rules
- 建立 Hooks 最小原型
- 建立 current-task / handoff / memory 模板

### 第二阶段：真实需求验证（下一步）
- 选一个真实前端或后端需求
- 走一遍完整黄金路径：需求 → 方案 → 契约 / 代码 → 校验 → 沉淀
- 根据反馈收敛 Rules 与 Hooks

### 第三阶段：基于真实需求继续沉淀
- 基于真实需求逐步沉淀常用 SOP，而不是预先一次性补齐模板
- `MEMORY.md` 拆分为索引 + 主题文件
- 结合真实任务正式落成工作模式：`Review-only` / `Implement` / `Fast-fix`

### 第四阶段：扩展增强
- 自动同步 issue / PR 摘要
- 接入工蜂 / Git 自动化链路（提交分支、创建 / 更新 MR、查询评审状态）
- 更强的多 Agent 协作
- 评估是否需要引入框架层

## 9. 为什么现在先沉淀文档

现在把方案沉淀成文档，有三个直接价值：

1. **避免重复生成**：后续继续设计或实施时，不需要重新把背景和结论再说一遍
2. **避免方向漂移**：任何后续改动都能回到这份方案，看是否仍符合初始设计原则
3. **方便协作与 handoff**：其他 AI / 同事可以直接基于这份文档理解目标、边界和路线图

## 10. 当前推荐结论

### 推荐表述
今后对外或对内描述时，优先使用：

**我们正在建设一套面向当前业务项目的「AI 原生研发闭环体系」：基于 Rules、Hooks、Memory、MCP 和契约驱动工作流，让 AI 参与从需求分析、方案设计、代码生成到测试验证和经验沉淀的完整研发闭环。**

### 当前路线判断
- **短期**：继续沿 `CodeBuddy` 原生能力推进
- **中期**：用真实需求验证闭环、补齐 SOP 和 Memory 拆分
- **长期**：仅在原生能力明显不足时，再评估引入 `OpenHarness` 一类框架

---

## 11. 后续维护建议

每次重大方案调整时，优先更新这份文档，再同步摘要到 `MEMORY.md`。这样 `MEMORY.md` 负责“索引与结论”，而这份文档负责“完整方案与设计依据”。
