# AI 原生研发闭环体系方案

> 版本：v0.2  
> 更新日期：2026-04-06  
> 适用范围：当前工作区四个项目（旧前端 `payclient-oa-depositmisview`、旧后端 `lqp`、新前端模板 `queryopenorderlist`、新后端模板 `xdc_wxpay`）

---

## 1. 方案目标

构建一套面向当前业务项目的 **AI 原生研发闭环体系**，让 AI 不只负责生成代码，而是在明确的阶段化流程约束下，参与并推进从需求分析到经验沉淀的完整研发闭环。

### 一句话目标

**通过 Rules + Hooks + Memory + MCP + 阶段化工作流，把 AI 从"代码补全工具"提升为"可治理的研发执行体"——每个阶段有固定输入、明确约束和标准交付物，过程可追溯、结果可验证、经验可沉淀。**

---

## 2. 设计背景

### 2.1 项目现状

这是一个**全新项目**，当前处于从零搭建阶段。工作区内有 4 个项目：

- **新前端模板** `queryopenorderlist`：Vue 3 + Vite + TDesign + XPage，是前端目标架构
- **新后端模板** `xdc_wxpay`：Node.js + XDC Node SDK + OpenAPI 契约，是后端目标架构
- **旧前端** `payclient-oa-depositmisview`：Vue 2 + Element UI，仅作迁移参考
- **旧后端** `lqp`：Egg.js / Koa / Kite，仅作迁移参考

新需求开发将在新模板上进行，旧项目只用于理解历史逻辑，不作为开发目标。

### 2.2 为什么需要这套体系

当前 AI 编码工具已经在用，但如果只是"把需求丢给 AI，聊几轮，拿到代码"，会遇到三个核心问题：

**问题一：AI 没有长期记忆，每次从零开始。**  
AI 不知道我们的内部框架（XPage / XDC / XContract）、业务约束、编码规范和历史坑点。它就像一个"每天都失忆的天才应届生"——基础知识极强，但对我们的系统一无所知。（参考 `voucher` 团队总结的"超级应届生"问题定义：全仓搜索超时、找错文档方向、同类规范问题反复出现、编译修复没有章法、边界遗漏反复发生。）

**问题二：信息不沉淀，质量靠个人。**  
同一个需求不同人跟 AI 聊，产出质量差异大，取决于"会不会提问"。做完后没有结构化产物，只有代码和零散聊天记录，没法审查也没法复用。（参考 `KM` 团队统计：18 人团队 288 需求/H1，41% 迁入率，信息不沉淀、过程不可追溯是核心痛点。）

**问题三：缺乏阶段化约束，AI 容易走最短路径。**  
没有明确的阶段边界和交付物要求，AI 会跳过需求分析直接写代码、跳过测试设计直接提交、跳过风险评估直接上线。（参考 `agent-skills` 的"反合理化机制"：AI 会说"这个很简单不用 spec""先写完再补测试"。）

**因此，我们需要的不是"让 AI 写更多代码"，而是"给 AI 套上约束——固定输入 → 约束规则 → 明确交付物"，让过程和结果能被持续约束和验证。**

## 3. 设计依据与参考来源

本方案的设计依据分为两个维度：**内容来源**（方案设计的是什么）和**落地方式**（方案怎么变成 AI 可执行的指令）。

**内容来源（4 层，决定方案设计的是什么）：**

| 层 | 来源 | 产出 |
|---|------|------|
| 1. 自身基础 | 四项目现状 + XPage / XDC / XContract 技术边界 + CodeBuddy 原生能力 | **约束条件**——能做什么、不能做什么 |
| 2. 同业务实践 | KM 团队（流程编排）+ voucher 团队（知识体系） | **流程和结构设计**——阶段怎么分、交付物怎么定、人机怎么分工 |
| 3. 业内优秀实践 | agent-skills + Superpower + CE + gstack | **具体机制**——反合理化、完成前验证、知识复利、前提挑战、任务粒度 |
| 4. Harness 思想 | Harness Engineering + OpenHarness | **认知坐标**——约束 + 反馈 + 控制系统的统一框架 |

**落地方式（参考 OpenHarness 的写法，把方案翻译成 AI 可执行指令）：**

| 载体 | 面向 | 回答的问题 | 写法参考 |
|------|------|-----------|---------|
| 方案文档（本文件） | 人 | 为什么这么设计 | — |
| Rules | AI | 你该怎么干活 | OpenHarness 的 `CLAUDE.md`：祈使句、精简、可执行 |
| Hooks | AI 运行时 | 哪些事情不许干 | OpenHarness 的 `PreToolUse` / `PostToolUse` |
| Memory | AI 上下文 | 之前学到了什么 | OpenHarness 的索引 + 主题文件拆分 |

> **一句话**：4 层来源决定"方案内容是什么"，OpenHarness 决定"方案怎么落地成 AI 可执行的指令"。

### 3.2 吸收策略与筛选原则

后续吸收外部经验时，按三层看材料：

1. **业务内实践层**：优先用来判断流程是否贴合真实业务研发场景
2. **通用工程方法层**：优先用来借鉴 Skill 结构、验证门禁、流程表达方式
3. **底层思想层**：优先用来统一认知坐标，不直接作为方案命名或落地模板

### 3.3 外部 Harness Skill 对照分析（2026-04-06 新增）

三个主流 Harness Skill 框架（Superpower、CE、gstack）解决同一个核心问题——**AI Agent 在执行任务时容易跳过规划、质量不稳定、缺乏固定 workflow 约束**——但思维模型和侧重点不同：

| 维度 | Superpower (obra) | CE (EveryInc) | gstack (garrytan/YC) |
|------|-------------------|---------------|---------------------|
| **核心思维** | 流程纪律 | 知识复利 | 产品打磨 + 角色模拟 |
| **工作流** | spec → plan → 任务拆解 → TDD 执行 → review → 提交 | brainstorm → plan → work → review → compound | office-hours → CEO review → eng review → 执行 → review → ship |
| **最强设计** | TDD 强制红绿循环 + 子代理两阶段审查（先规范合规，再代码质量） + 完成前验证（禁止无证据声称完成） | `/ce:compound` 做完即沉淀 + 80/20 原则（80% 精力在计划和审查） | `/office-hours` 苏格拉底式灵魂拷问 + `/learn` 跨会话累积学习（按 Patterns/Pitfalls/Preferences/Architecture 分类） + 跨模型交叉审查 |
| **最弱点** | 小任务流程太重，授权频繁、token 消耗大 | 改进建议容易不痛不痒 | 偏产品/创业场景，工程纪律不如 Superpower |
| **典型使用组合** | 大工程的编码执行阶段 | 持续迭代项目的知识沉淀 | 大工程的需求打磨 + 产品方向验证 |

#### 已吸收到本方案的设计点

| 吸收点 | 来源 | 在方案中的体现 |
|--------|------|--------------|
| **完成前验证必须有证据** | Superpower `verification-before-completion` | 全流程每个阶段的"验证与门禁"——不允许 AI 说"应该能过"或"大概完成了"，必须运行命令、读取输出、确认结果后才能声称完成 |
| **反合理化机制** | Superpower + `agent-skills` | 编码阶段"逐任务执行，不跳步"约束——AI 不能说"太简单不用分步""先写完再补测试" |
| **设计前强制 brainstorming** | Superpower `brainstorming` | 需求分析阶段的结构化穷举 + 方案设计阶段的多方案对比——不允许跳过设计直接写代码 |
| **任务粒度控制** | Superpower `writing-plans`（每任务 2-5 分钟） | 编码阶段的任务拆解推荐小粒度，每个任务有明确的文件路径、代码和验证步骤 |
| **做完即沉淀（知识复利）** | CE `compound` | 归档阶段的 Rules / Memory / 测试资产更新——不是"做完就结束"，而是"做完就提炼" |
| **80/20 计划审查原则** | CE | 人机分工模型中"设计阶段人主导"的理论支撑——大部分精力应在计划和审查上，编码执行只占 20% |
| **需求阶段主动挑战假设** | gstack `office-hours` | 需求分析阶段的穷举式补全机制——不只是被动提取需求，也应主动追问"这是对的问题吗？""如果什么都不做会怎样？" |
| **学习记录分类管理** | gstack `learn`（Patterns / Pitfalls / Preferences / Architecture） | Memory 层的长期知识分类方向——不是一个大杂烩的 MEMORY.md，而应按模式、陷阱、偏好、架构分类沉淀 |
| **安全护栏** | gstack `careful` / `freeze` / `guard` | Hooks 层 `pretool_guard` 的运行时风险控制——高危命令拦截、敏感路径保护 |

#### 关注但当前不急于吸收的设计点

| 设计点 | 来源 | 原因 |
|--------|------|------|
| **TDD 强制红绿循环**（先写失败测试 → 最少代码通过 → 删除测试前代码） | Superpower | 当前项目测试基础设施未建立，过早强制 TDD 会卡住。计划在第三阶段评估引入 |
| **子代理两阶段审查**（先查规范合规，再查代码质量） | Superpower | 需要多 Agent 支持，当前 CodeBuddy Team 模式可探索但不是第一优先级 |
| **跨模型交叉审查**（Claude + Codex 独立审查同一分支） | gstack | 思路超前但当前不具备多模型条件 |
| **发散式改进发现**（`/ce:ideate`） | CE | 改进点应来自真实需求验证而非 AI 发散 |
| **工程回顾报告**（`/retro` 按周生成提交分析 + 代码质量指标 + 趋势跟踪） | gstack | 有价值但当前需求量不足以支撑周回顾 |

筛选时默认问四个问题：

- 它解决的是我们**哪个真实问题**？
- 它属于**哪一层**参考来源？
- 它是否适配当前 `CodeBuddy + Rules / Hooks / Memory / MCP + 契约驱动` 路线？
- 它应该进入**正式方案、Rules、Memory**，还是仅保留参考？

## 4. 全流程设计

### 4.1 流程全景与人机分工模型

一个新需求从进来到上线归档，分为 **6 个阶段**，每个阶段有明确的人机分工：

```
需求与分析 → 规范定义(spec) → 方案设计 → 编码实现 → 测试与验证 → 归档与沉淀
                                                                    ↓
                                                              反馈闭环（反哺下一轮）
```

#### 人机分工总原则

| 阶段 | 人的角色 | AI 的角色 |
|------|---------|----------|
| 需求与分析 | **主导**：确认业务目标、补充隐性约束、审查分析结果 | **辅助**：结构化需求、提取功能点与规则、识别风险 |
| 规范定义(spec) | **主导**：审查 spec 各章节、确认目标和接口定义 | **辅助**：生成 spec 草案（目标+接口+文件规划+测试策略+约束） |
| 方案设计 | **主导**：架构决策、关键取舍、审查任务拆解 | **辅助**：梳理现状、生成任务清单 |
| 编码实现 | **辅助**：审查代码、补充领域知识、确认关键逻辑 | **主导**：按任务清单逐步生成代码，契约先行 |
| 测试与验证 | **辅助**：确认测试前置条件、验收最终结果 | **主导**：生成测试用例、执行测试、根据报错修正 |
| 归档与沉淀 | **主导**：确认沉淀内容的准确性和长期价值 | **辅助**：从过程中提炼规则更新、记忆更新、模板更新 |

> **设计理由**：这个人机分工模型参考了 `voucher` 团队的实践——"设计阶段人主导 AI 辅助梳理现状；开发测试阶段 AI 主导、人辅助审查补充"。他们发现：方案设计需要业务判断和架构决策，AI 在这一步更适合辅助而非主导；而编码和测试是规则性更强的工作，AI 在有结构化上下文的情况下可以稳定输出。

#### 假设场景

后续每个阶段将用以下典型场景来具象化说明：

> **在 `queryopenorderlist` 上新增一个"待处理订单查询"页面，对应在 `xdc_wxpay` 上新增查询接口，接口需调用下游 Svrkit 服务获取订单数据。**

### 4.2 需求与分析阶段

#### 这个阶段解决什么问题

如果直接把产品需求描述丢给 AI 让它写代码，AI 会脑补接口字段、遗漏业务规则、忽略异常流程。`KM` 团队的实践数据表明：在他们的 SMS 案例中，PRD 明确写出的规则只有 6 条，AI 通过结构化穷举补全了 5 条（45%）——这些遗漏的规则如果到编码阶段才发现，返工成本远高于前置分析。

#### 人机分工

- **人**：确认业务目标、补充 PRD 未写明的隐性约束、审查 AI 产出的分析结果
- **AI**：结构化提取功能点、业务规则、风险项；对每条规则做四要素结构化；**主动挑战前提假设**（这是对的问题吗？如果什么都不做会怎样？现有代码中有哪些可以复用的？）

#### 输入 → 约束 → 交付物

| 输入 | 约束 | 交付物 |
|------|------|--------|
| 产品需求描述（TAPD / 口头 / 文档） | 每条业务规则必须结构化为四要素：规则描述、满足时行为、违反时行为、边界条件 | `requirement-analysis.md`：功能点清单 + 业务规则（四要素） + 风险初评 |

#### 场景示例

> **"待处理订单查询"需求进来后：**
> - AI 先做前提挑战：现有 `payclient-oa-depositmisview` 中是否已有类似查询页面可复用逻辑？如果不做这个功能，用户当前怎么解决？ → 确认确实需要新建
> - AI 从需求描述中提取功能点：查询条件（时间范围、订单状态、商户号）、结果分页、导出功能
> - AI 对每个功能点穷举业务规则：时间范围上限（如最多 90 天）、分页大小限制、导出最大条数、无结果时的提示
> - 人确认：补充"只能查询本组织下的订单"这条 PRD 未写明的权限约束
> - 产出 `requirement-analysis.md`，包含 5 个功能点、12 条业务规则（四要素格式）、3 条风险项

#### 设计理由

- **四要素结构化规则**：参考 `KM` 团队的实践——有了"规则描述、满足时、违反时、边界条件"四个要素，后续测试用例可以机械推导，而不是靠经验判断。
- **穷举式补全**：参考 `KM` 团队的五种策略（CRUD 补全 / 正反面 / 边界值 / 异常流 / 交叉影响），主动探查 PRD 未写明的隐藏规则。
- **主动挑战假设**：参考 `gstack` 的 `/office-hours` 机制——不只是被动提取需求，还应主动追问三个问题：(1) 这是对的问题吗？(2) 如果什么都不做会怎样？(3) 现有代码中有哪些可以复用的？`gstack` 的实践表明，在需求阶段加入"前提挑战"（Premise Challenge）环节，可以显著减少后续返工。

### 4.3 规范定义阶段（Spec）

#### 这个阶段解决什么问题

需求分析产出了功能点和业务规则，但如果直接让 AI 写代码或拆任务，AI 会自己决定用什么接口形态、文件放哪里、错误怎么处理——这些决定没经过人确认，写完才发现"不是我想要的"→ 返工。`agent-skills` 总结的核心观察是：**"Code before spec is the #1 cause of wasted work."** Superpower 和 gstack 也有一致结论——无论任务多简单，都必须先有 spec 再写代码。

#### 人机分工

- **人**：审查 spec 各章节（目标、接口、文件规划、测试策略、约束），确认"做什么"是对的
- **AI**：基于 `requirement-analysis.md` 生成 `spec.md` 草案

#### 输入 → 约束 → 交付物

| 输入 | 约束 | 交付物 |
|------|------|--------|
| `requirement-analysis.md` + 项目 Rules + 现有代码 | spec 必须包含 5 个必选章节；禁止"TBD""待定"等占位符；本阶段只定义"做什么"，不拆任务、不写代码 | `spec.md`：目标 + 接口定义 + 文件规划 + 测试策略 + 约束与边界 |

#### `spec.md` 的 5 个必选章节

1. **§1 目标（Goals）**：一句话描述本需求要实现什么 + 成功标准
2. **§2 接口定义（API / Interface）**：新增或修改的接口路径、方法、请求参数、响应结构、错误码；前端需求则为页面路由、组件层次、数据流
3. **§3 文件规划（File Plan）**：将创建的新文件及其职责 + 将修改的已有文件及修改点
4. **§4 测试策略（Test Strategy）**：基于业务规则列出需覆盖的测试维度（正向/反向/边界/权限）
5. **§5 约束与边界（Constraints）**：技术约束、业务约束、明确标注"不做什么"（Out of Scope）

#### 场景示例

> **"待处理订单查询" spec 阶段：**
> - AI 生成 `spec.md`：
>   - §1 目标：在运营后台新增"待处理订单查询"功能，支持按时间范围、状态、商户号查询，结果分页展示
>   - §2 接口：`GET /api/pending-orders`，请求参数（`start_date`、`end_date`、`status`、`page`、`page_size`）、响应结构（`total`、`list[]`）、错误码（参数校验失败、无权限、下游超时）
>   - §3 文件规划：后端新增 `pending-orders-controller.ts`；前端新增 `pending-orders/` 页面目录（Page + Model + Service）+ 路由注册到 `src/index.ts`
>   - §4 测试策略：正向（正常查询）、反向（超 90 天）、边界（分页极值）、权限（非本组织）
>   - §5 约束：不做导出功能（Out of Scope）、下游超时需降级处理
> - 人审查：确认接口定义和文件规划，补充"只能查询本组织下的订单"

#### 设计理由

- **spec 独立于 design**：参考 `agent-skills`（Define ≠ Plan）、`Superpower`（brainstorming ≠ writing-plans）、`gstack`（office-hours ≠ eng-review）三方一致结论——"定义做什么"和"规划怎么做"是两个不同的决策，不应该混在一起审查。
- **spec 是后续所有阶段的唯一锚点**：design 基于 spec 拆任务，coding 基于 spec 知道每个文件该做什么，testing 基于 spec 知道该验证什么，review 基于 spec 检查代码是否符合预期。
- **禁止占位符**：参考 `Superpower` 的 `writing-plans`——"Never use placeholders: TBD, TODO, implement later"。spec 里的每一项都必须是确定的。

### 4.4 方案设计阶段

#### 这个阶段解决什么问题

spec 定义了"做什么"，但还需要一层**任务级的结构化上下文桥梁**，才能稳定驱动代码生成。`voucher` 团队的核心公式是：`任务提案 × 组件/框架知识 × 领域知识 × 规范 → 高质量业务代码`。

#### 人机分工

- **人**：审查任务拆解的合理性和粒度
- **AI**：梳理系统现状（通过代码和 MCP）、基于 spec 生成任务清单、辅助风险识别

#### 输入 → 约束 → 交付物

| 输入 | 约束 | 交付物 |
|------|------|--------|
| `spec.md` + 项目 Rules + 现有代码 | 任务清单中契约任务排最前；所有任务必须可追溯到 `spec.md` 中的某一项 | ① `tasks.md`：按章节组织的任务清单<br>② `design-notes.md`（按需）：关键技术决策与风险评估 |

#### 场景示例

> **"待处理订单查询"设计阶段：**
> - AI 基于 `spec.md` 生成 `tasks.md`：
>   - §0 契约定义：在 `.xdc/contract/openapi.yaml` 中新增 `GET /api/pending-orders` 接口定义
>   - §1 后端实现：新增 `pending-orders-controller.ts` + 下游 Svrkit 调用
>   - §2 前端页面：新增查询页面组件 + 路由注册 + Model/Service
>   - §3 联调验证：前后端对接确认
> - 人审查任务拆解，确认粒度和顺序

#### 设计理由

- **契约先行**：参考 `KM` 团队的做法——`tasks.md` 中契约任务排第 0 章，AI 不用脑补接口形态。这与我们的 `XDC / XContract` 机制天然匹配：先定义 `openapi.yaml`，再生成代码骨架。
- **任务提案作为 AI 执行上下文**：参考 `voucher` 团队的 SDD 提案思想——提案不是给人看的需求文档，而是给 AI 看的结构化上下文。每个任务包含具体流程逻辑、领域约束、组件使用方式和规范要求。
- **风险评估按需独立化**：参考 `KM` 团队的做法——安全评估（STRIDE）、资金风险等做成独立交付物。我们当前阶段先以 `design-notes.md` 统一承载关键决策和风险，复杂需求再拆分。

### 4.5 编码实现阶段

#### 这个阶段解决什么问题

有了 spec 和任务清单，AI 就有了明确的执行上下文。但如果没有项目规范约束，AI 仍然会用错框架调用方式、违反目录结构、产出不一致的代码风格。`voucher` 团队总结的翻车案例很典型：AI 用了错误的框架调用方式、漏掉关键业务校验、不知道该调哪个内部组件、代码风格不符合规范。

#### 人机分工

- **人**：审查 AI 生成的代码、补充 AI 看不到的领域知识、确认关键逻辑正确性
- **AI**：按 `tasks.md` 逐任务实现代码，遵循 Rules 中的项目约束

#### 输入 → 约束 → 交付物

| 输入 | 约束 | 交付物 |
|------|------|--------|
| `tasks.md` + `spec.md` + 项目 Rules + 现有代码 | ① 契约先行：先更新 `openapi.yaml`，再写实现<br>② 遵循 Rules：前端走 Pages↔Models↔Services 三层分离，后端 Controller 以生成结构为准<br>③ 逐任务提交，不跳步；每任务须包含具体文件路径、代码和验证步骤，禁止"TBD""TODO"等占位符<br>④ 完成前必须出示证据：运行验证命令 → 读取输出 → 确认通过后才可声称完成，禁止"应该""大概"等词汇 | 符合契约定义的前后端代码，可通过 lint 和基础编译检查 |

#### 场景示例

> **"待处理订单查询"编码阶段：**
> - AI 先执行 §0：在 `openapi.yaml` 中新增 `/api/pending-orders` 接口定义
> - AI 执行 §1：在 `xdc_wxpay` 中新增 `pending-orders-controller.ts`，按契约定义实现请求参数校验、下游 Svrkit 调用、响应组装；业务异常使用 `BizError` + 标准错误码
> - AI 执行 §2：在 `queryopenorderlist` 中新增 `pending-orders` 页面，在 `src/index.ts` 注册路由，按 Pages↔Models↔Services 分层实现；API 调用使用契约生成客户端
> - 人审查：确认下游调用的超时处理、分页逻辑、权限校验是否正确

#### 设计理由

- **Rules 即规范手册**：参考 `voucher` 团队的推导——AI 不知道的规范就会反复违反，应该写成 AI 可读的 Rules 文件自动加载，而不是靠人在 Review 里反复纠正。我们已落地的 5 条 Rules（`workspace-architecture`、`xpage-frontend-guardrails`、`xdc-backend-contract-guardrails`、`delivery-workflow`、`git-workflow`）正是这个思路的实现。
- **逐任务执行，不跳步**：参考 `agent-skills` 的反合理化机制 + `Superpower` 的任务粒度控制（每任务 2-5 分钟）——AI 容易说"先写完再补测试""这个很简单不用分步"。`Superpower` 进一步要求每个任务必须包含**具体的文件路径、完整代码和验证步骤**，禁止使用"TBD"、"TODO"等占位符，禁止"与任务 N 类似"这种模糊引用。
- **完成前必须出示证据**：参考 `Superpower` 的 `verification-before-completion` 机制——在声称任何步骤完成之前，必须：识别验证命令 → 运行命令 → 读取完整输出 → 确认输出是否支撑结论。禁止使用"应该""大概""看起来"等词汇。这个机制将在编码和测试阶段的"验证与门禁"中体现。
- **Hooks 运行时守护**：编码阶段通过 `PreToolUse` Hook 拦截高危命令（`rm -rf`、`git push --force`）和敏感路径写入，防止 AI 在执行中产生不可逆操作。参考 `gstack` 的 `careful`/`freeze`/`guard` 三级安全护栏思路，我们的 `pretool_guard.py` 承担了类似职责。

### 4.6 测试与验证阶段

#### 这个阶段解决什么问题

代码写完后，如果没有前置的测试设计，测试覆盖度完全依赖编码者的经验。`KM` 团队的做法是把测试左移到编码前：基于需求分析阶段的四要素结构化规则，测试用例可以机械推导——满足时 → 正向用例，违反时 → 反向用例 + 错误码验证，边界条件 → 边界值测试。他们的 SMS 案例产出了 42 条用例（30 AT + 6 IT + 6 E2E），编码前就已就绪。

#### 人机分工

- **人**：确认测试前置条件、验收最终结果、处理 AI 解决不了的环境问题
- **AI**：基于业务规则生成测试用例、编写测试脚本、执行测试、根据报错自动修正

#### 输入 → 约束 → 交付物

| 输入 | 约束 | 交付物 |
|------|------|--------|
| `requirement-analysis.md`（业务规则四要素）+ 已实现代码 + 契约定义 | 测试用例必须覆盖所有业务规则的正向/反向/边界场景；契约校验作为验证前置条件；**每次声称测试通过必须附带命令输出证据**（退出码 + 通过/失败数），不接受"应该能过""之前跑过了" | ① `test-cases.md`：结构化测试用例<br>② 通过 lint、编译、契约校验的验证结果 |

#### 场景示例

> **"待处理订单查询"测试阶段：**
> - AI 基于 12 条业务规则生成测试用例：
>   - 正向：正常时间范围 + 有效状态 → 返回分页结果
>   - 反向：时间范围超 90 天 → 返回参数校验错误码
>   - 边界：分页大小为 0 / 最大值 / 最大值+1
>   - 权限：非本组织订单 → 返回无权限错误码
> - AI 执行 lint 检查和契约校验（`openapi.yaml` 与实际接口是否一致）
> - 人验收最终结果，确认关键场景覆盖

#### 设计理由

- **测试左移**：参考 `KM` 团队的核心方法——测试用例在编码前就基于需求分析的结构化规则生成，而不是写完代码再补。四要素结构化规则让"这条规则应该写几条用例"变成确定性问题。
- **完成前验证必须有证据**：参考 `Superpower` 的 `verification-before-completion`——测试阶段是最容易出现"虚假完成"的环节。AI 声称"测试通过"时必须附带命令输出（退出码 + 通过/失败数），不接受"应该能过""之前跑过了"等推断。`Superpower` 的实践表明：24 次失败记忆中，多数源于未做独立验证就声称完成。
- **测试资产即反馈循环**：参考 `voucher` 团队的推导——没有测试用例告诉 AI"什么情况该通过、什么情况该拒绝"，AI 就会在边界上反复踩坑。测试资产本身就是知识体系的一部分，会随着迭代不断积累。

### 4.7 归档与沉淀阶段

#### 这个阶段解决什么问题

需求做完后，如果不做沉淀，下一次需求又从零开始。`voucher` 团队强调的"知识飞轮"核心就是这一步：提案中沉淀的业务规则 → 更新领域知识文档；发现的可复用模式 → 进化为新的 Skill/Rule；过时的中间知识 → 淘汰清理。每转一圈，知识体系就厚一层。

#### 人机分工

- **人**：确认沉淀内容的准确性、长期价值，避免把临时讨论误沉淀为长期知识
- **AI**：从过程产物中提炼规则更新、记忆更新、模板更新

#### 输入 → 约束 → 交付物

| 输入 | 约束 | 交付物 |
|------|------|--------|
| 本次需求的全部过程产物 + 现有 Rules / Memory | 只沉淀长期有效、可复用的经验；临时讨论不写入 MEMORY；沉淀内容按四类归档：**Patterns**（可复用模式）、**Pitfalls**（踩坑记录）、**Preferences**（偏好约定）、**Architecture**（架构决策） | ① Rules 更新（如果发现新的约束或坑点）<br>② Memory 更新（按四类分类的长期经验摘要）<br>③ 模板更新（如果发现可复用的任务模式） |

#### 场景示例

> **"待处理订单查询"归档阶段：**
> - AI 按四类整理沉淀内容：
>   - **Pattern**：分页查询的参数校验逻辑可以复用 → 记录到 Memory 的 Patterns 分类
>   - **Pitfall**：下游 Svrkit 调用超时需要做降级处理，这是一个通用坑点 → 建议补充到 `xdc-backend-contract-guardrails` Rules 中
>   - **Preference**：查询接口命名统一用 `GET /api/{resource-name}`，不用 `POST /api/query{ResourceName}` → 记录到 Preferences 分类
> - 人确认更新内容准确，避免把本次特殊处理误记为通用规则

#### 设计理由

- **反馈闭环**：参考 `voucher` 团队的"设计 → 开发 → 测试 → 提炼 → 下一次设计"完整闭环。每次需求做完不是结束，而是知识体系的一次迭代。`CE (Compound Engineering)` 把这个理念总结为 `/ce:compound`——做完即沉淀，将审查和工作中的经验转化为可重用资产。
- **知识分类管理**：参考 `gstack` 的 `/learn` 命令——学习记录按 Patterns（可复用模式）、Pitfalls（踩坑记录）、Preferences（偏好约定）、Architecture（架构决策）四类管理，支持搜索、清理过时条目、导出到项目文档。这为我们后续 Memory 从单一 `MEMORY.md` 拆分为按主题分类的索引 + 主题文件提供了具体的分类模型。
- **代码第一性**：参考 `voucher` 团队的原则——能从代码和 MCP 获取的信息不重复堆进 Rules 和知识文档。Rules 只记录"代码看不出来但 AI 必须知道"的东西，如平台约束、历史坑点、宿主行为、禁忌做法。

### 4.8 反馈闭环

全流程不是线性单次执行，而是一个持续收敛的循环：

```
需求分析 → spec → 设计 → 编码 → 测试 → 归档
                                  ↓
                    Rules / Memory / 测试资产 / 模板
                                  ↓
                            下一个需求（起点更高）
```

每次需求完成后，归档阶段产出的更新会直接改善下一次需求的执行质量：

- **Rules 更准**：新发现的约束和坑点被补进规则，AI 下次不再犯同样的错
- **Memory 更厚**：长期经验积累，AI 的起始上下文更丰富
- **测试资产更全**：边界条件不断补充，测试覆盖度持续提升
- **模板更成熟**：可复用的任务模式被提炼为标准模板，效率递增

> **这就是 `voucher` 团队所说的"飞轮效应"**：第一个需求最慢，因为知识体系从零开始；后续同类需求越来越快，因为前面积累的 Rules、Memory、测试资产和模板都在复利。
>
> `CE (Compound Engineering)` 把这个理念总结为"80/20 原则"——80% 精力投入在计划和审查（设计阶段人主导），20% 投入在执行（编码阶段 AI 主导）。每一次循环都会产生复利：brainstorm 完善 plan，plan 指导 work，review 捕捉问题，compound 编纂知识，循环自强化。
>
> `gstack` 的 `/learn` 命令进一步提供了**知识分类管理**的参考：按 Patterns（模式）、Pitfalls（陷阱）、Preferences（偏好）、Architecture（架构）四类管理学习记录，支持搜索、清理过时条目、导出到项目文档。这为我们后续 Memory 拆分（从单一 MEMORY.md 到按主题分类的索引 + 主题文件）提供了具体的分类模型。

## 5. 体系支撑能力

全流程设计依赖 6 层底座能力，当前均基于 `CodeBuddy` 原生能力实现：

### A. Rules 层 — 稳定背景与开发约束

| Rule | 适用范围 | 核心作用 |
|------|---------|---------|
| `workspace-architecture` | 整个工作区 | 识别当前项目 + **结构化项目配置表**（供 Skill 引用）+ 契约对齐优先 |
| `xpage-frontend-guardrails` | 新前端 `queryopenorderlist` | 路由真相源、平台壳约束、三层分离、API 调用方式 |
| `xdc-backend-contract-guardrails` | 新后端 `xdc_wxpay` | 契约先行、Controller 生成结构、BizError 标准错误码 |
| `delivery-workflow` | 所有任务 | 任务分类（Implement / Fast-fix / Review-only）+ **6 阶段 Skill 链**调度 + 通用约束 |
| `git-workflow` | 涉及 Git 操作时 | 分支命名规范 + Commit 规范 + 多仓操作指引 |

### B. Skills 层 — 阶段化执行指令

| Skill | 阶段 | 核心产出 |
|-------|------|---------|
| `requirement-analysis` | 1. 需求与分析 | `requirement-analysis.md` |
| `spec` | 2. 规范定义 | `spec.md`（含场景判断：纯前端/纯后端/全栈） |
| `design` | 3. 方案设计 | `tasks.md` |
| `coding` | 4. 编码实现 | 编码前契约一致性校验 + 符合 spec 的前后端代码 |
| `testing` | 5. 测试与验证 | `test-cases.md` + 验证结果 |
| `archiving` | 6. 归档与沉淀 | Rules / Memory / 模板更新 |
| `ui-guide` | 按需（前端页面开发时） | XPage 6 种低代码模板 + TDesign 组件速查 + 代码模式 |

> AI 根据当前阶段自动加载对应 Skill，每个阶段完成后需人确认才进入下一阶段。

### C. Commands 层 — 用户快捷入口

| 命令 | 作用 |
|------|------|
| `/implement [需求描述]` | 启动 Implement 全流程（6 阶段） |
| `/fast-fix [问题描述]` | 启动 Fast-fix 模式（跳过需求分析、spec 和设计，直接编码→验证→归档） |

### D. Hooks 层 — 运行时治理与风险控制

| Hook | 触发时机 | 核心作用 |
|------|---------|---------|
| `session_start_context` | 会话开始时 | 自动识别当前项目，注入对应 Rules 和上下文提示 |
| `pretool_guard` | 每次工具调用前 | 拦截高危命令（`rm -rf` / `git push --force`）、保护 `.git` 和 `.codebuddy` 目录、超出工作区的操作需人工确认 |
| `post_lint_check` | 文件写入/编辑后 | 自动提醒检查 lint 结果，确认无新增错误 |

### E. Memory / Context / Template 层 — 长期知识与任务上下文

| 组件 | 作用 |
|------|------|
| `MEMORY.md` | 长期有效的项目知识、方案摘要、经验索引 |
| `context/current-task.md` | 当前任务的临时上下文，会话间可续接 |
| `templates/handoff-template.md` | 跨会话任务交接模板 |
| `templates/memory-capture-template.md` | 经验沉淀模板（四分类） |
