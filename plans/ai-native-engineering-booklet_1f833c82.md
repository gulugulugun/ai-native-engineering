## 产品概述

将现有的 AI 原生研发闭环体系方案（v0.2）梳理为一本多文件结构的技术小册，目标是让自己（以及后续团队成员）直观、清晰地理解这套架构的每一个设计点：它来自哪里、参考了什么、为什么这样设计、好处是什么。小册同时作为后续架构调整的决策依据。

## 核心特征

- **多文件小册结构**：目录页 + 各章独立 Markdown 文件，支持渐进式阅读
- **设计溯源贯穿全文**：每个设计点都标注「来自哪里 -> 解决什么问题 -> 好处是什么」三段式推理链
- **架构图示辅助理解**：在关键位置使用 Mermaid 图、引用已有 PNG 图片，让体系结构一目了然
- **标准深度**：15000-25000 字，完整讲清设计理念、全流程、体系支撑、参考来源
- **案例章节留占位**：体系部分先写完，落地案例等真实需求跑完再补充
- **先内部后外部**：当前版本服务于自己理解和架构调整，后续可改写为对外分享版

## 技术栈

- 输出格式：Markdown（多文件结构）
- 图表：Mermaid 语法（流程图、架构图、序列图）
- 图片引用：已有 PNG 图片通过相对路径引用
- 目录管理：`README.md` 作为小册入口与导航

## 实现方案

### 总体策略

以现有方案文档 `ai-native-engineering-scheme.md` 为主干内容来源，以设计溯源表 `design-provenance.md` 为「来自哪里」的追踪依据，以 3 篇参考文章为写作风格参考，以 Skills/Rules/Hooks/Agents 的实际源码为准确性校验依据，重新组织为面向阅读理解的小册结构。

### 写作原则

融合三篇参考文章的优点：

- **学 cavanwan**：推理链叙事（翻车案例 -> 推导需要什么 -> 解法），每个设计决策都有「为什么」
- **学 tobytang**：案例+数据展示方式（产出目录树、时间线对比），为案例章节预留格式
- **学 zipsu**：可操作格式（输入 -> 约束 -> 交付物），让读者看完能直接照做

## 实施注意事项

1. **准确性优先**：所有 Skill 描述、Rule 内容、Hook 行为都以实际源码为准，不凭记忆描述
2. **溯源链完整**：每个设计点必须完成「来自哪里 -> 解决什么 -> 好处」三段式，缺一不可
3. **不过度美化**：「关注但当前不急于吸收的设计点」和「局限」也要诚实呈现
4. **图文配合**：Mermaid 图放在对应章节正文中，PNG 图片用相对路径引用
5. **信息缺失标注**：如果某处信息来源不够明确，在文中用 `<!-- TODO: 需要补充 -->` 标注，并在最终汇总给用户

## Agent Extensions

### Skill

- **drawio-diagrams**
- 用途：为小册生成关键架构图和流程图（如体系架构总图、6阶段流程图），补充 Mermaid 无法表达的复杂图表
- 预期产出：drawio 格式的架构图文件，可导出为 PNG 嵌入小册

### SubAgent

- **code-explorer**
- 用途：在编写各章节时，搜索 Skills/Rules/Hooks/Agents 的实际源码，确保小册中的描述与实现一致
- 预期产出：每个产物的准确描述、关键代码片段引用、功能行为的事实核验
