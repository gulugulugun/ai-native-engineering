## 文章结构

### 一、背景与问题（为什么要做）

> 开篇定调：不是"AI 多厉害"，而是"不治理 AI 会出什么问题"

- [ ] 团队现状数据（人力、需求量、项目构成）
- [ ] 核心问题定义（3 个，参考方案文档 §2.2）：
  1. AI 没有长期记忆，每次从零开始
  2. 信息不沉淀，质量靠个人
  3. 缺乏阶段化约束，AI 走最短路径
- [ ] 1-2 个真实翻车案例（学 cavanwan：用案例推导，不直接列结论）
- [ ] 一句话目标

### 二、设计理念（为什么这么设计）

> 不长，但要把"思路"讲清楚，让读者理解后续流程不是拍脑袋拍出来的

- [ ] 核心思路：固定输入 → 约束规则 → 明确交付物
- [ ] 四层参考来源（一张表，简要）
- [ ] 关键吸收点（完成前验证、反合理化、知识复利、验证回流）
- [ ] 人机分工总原则（设计人主导 / 编码AI主导）

### 三、全流程设计（做什么）

> 这是主体，但要紧凑。每阶段只讲"干什么、出什么、怎么检查"

- [ ] 流程全景图（mermaid，一张图看全貌）
- [ ] 6 阶段概览表（阶段 / Skill / 交付物 / 人机分工）
- [ ] 每阶段简要说明（每阶段 3-5 行，不展开细节）
- [ ] 3 个关键机制：
  - 契约先行（先定 openapi.yaml 再写代码）
  - 验证失败多级回流（5 种判定 + 4 级根因分类）
  - 做完即沉淀（归档阶段反哺 Rules/Memory）

### 四、体系支撑能力（有什么底座）

> 一张表够了，不展开

- [ ] Rules（5 条）/ Skills（7 个）/ Hooks（3 个）/ Memory / MCP 一览表
- [ ] 一句话说明每个组件的作用

### 五、落地案例 ← **最重要，等真实需求跑完填**

> 这一章决定文章的说服力。学 tobytang 的展示方式。

- [ ] **案例：[第一个真实需求名称]**
  - 需求背景（一段话）
  - 完整产出目录树（`tree` 格式）
  - 每阶段产出摘要（功能点数 / 规则数 / 任务数 / 测试用例数）
  - 时间线对比表（传统方式 vs 本工作流）
  - 过程中的关键决策（1-2 个，附推理）
  - 过程中踩的坑（1-2 个，附解法）
- [ ] （可选）案例二：更大/更复杂的需求

### 六、实践中踩过的坑

> 单独列出，增加真实感。学 tobytang 的写法。

- [ ] 坑 1：[描述] → 原因 → 解法 → 已沉淀到哪里
- [ ] 坑 2：...
- [ ] 坑 3：...

### 七、效果与局限

- [ ] 量化数据（交付物数量、时间对比、测试覆盖率等）
- [ ] 当前局限（诚实列出）
- [ ] 下一步计划

### 附录

- [ ] 参考来源总表（精简版，指向方案文档）
- [ ] 标签体系（如有）
