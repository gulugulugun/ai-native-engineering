## 补充 wxpay 平台 + XDC Kite 框架文档认知

读取了两篇 iWiki 文档，补充对新项目模板宿主平台和后端框架的理解：

### 文档1: wxpay 运营系统接入指南 (p/4012548245)
- 这是新前端模板部署到的**宿主平台**完整接入文档
- 关键发现：
  - **路由跳转必须用 `window.wxpay.router`，禁止直接用 vue-router** — 这是写 Rules 时必须强调的约束
  - **API 请求用 `window.wxpay.request`**（axios 风格，自带权限处理）或契约调用
  - 菜单英文名 = 开发模块名
  - 多菜单共用模块时，非创建菜单的路由需加 `/菜单英文名/` 前缀
  - 开发全流程：菜单创建 → 模板特性 → 开发模块 → AnyDev → 编码 → 测试 → 发布

### 文档2: XDC Node SDK (Kite 框架) 开发指南 (p/1595683571)
- 这是新后端模板 `xdc_wxpay` 使用的框架详细文档
- 关键发现：
  - 统一配置文件 `xdc.yml`，替代了 `.image.json` + `package.json` + `pb.config.js`
  - 自动加载 + 依赖注入（DI）：属性注入 vs 构造参数注入，默认单例
  - 异常处理：`BizError(errCode)` 标准方式
  - 服务调用：`xdc.yml` 配置 deps.modules → `npm run gen-client` 生成客户端代码
  - 动态配置：七彩石（Rainbow），默认 120s 更新
  - ⚠️ 属性注入在构造函数**之后**执行，构造函数内需用参数注入

## 补充：契约开发流程 + Clean Architecture 目录约定（图片）

用户提供了两张 XDC Kite 框架的关键图片：

### 图1: 契约开发流程（四步闭环）
1. 初始化选择契约框架 → 2. 在契约平台编辑接口契约 → 3. `npm run gen` 生成代码 → 4. 服务端填业务逻辑，客户端一行调用
- 核心：契约驱动 = 接口定义先行 → 代码骨架自动生成 → 只关注业务逻辑

### 图2: Clean Architecture 到目录结构映射
- 基于洋葱圈模型：Entity(业务规则) → Controller(用例) → Middleware(适配器) → 框架
- **controller 必须由契约生成结构**（函数签名），开发者只填业务逻辑
- **没有 service 目录**：轻量用 entity，复杂拆微服务
- Entity 按分层：应用层=AO，领域层=DO，领域DAO=PO

## Harness Engineering 学习 + OpenHarness 仓库分析

学习了 Harness Engineering 概念（OpenAI 提出的 AI 工程新范式）和港大开源的 OpenHarness 框架（11733 行 Python，44 倍轻于 Claude Code）。

### 关键结论
- Harness = 约束 + 反馈 + 控制系统，模型提供智能，Harness 提供手、眼、记忆和安全边界
- OpenHarness 有 10 个子系统：Engine/Tools/Skills/Prompts/Memory/Permissions/Hooks/Config/Coordinator/Plugins
- 与我们现有体系对比：大部分能力 CodeBuddy 原生已具备（Rules≈CLAUDE.md、Prompt 自动注入、截断防护、原生 Hooks）
- 值得继续关注的：按需技能加载，以及 Hooks 之上的更复杂治理链路（如多阶段审查、插件生态）
- 核心判断：**我们当前建设的是面向业务项目的 AI 原生研发闭环体系，它借鉴了 Harness Engineering 的思想，但正式定位不是构建一个通用 Harness；现阶段应优先验证 CodeBuddy 原生能力，而不是急着引入额外框架**
