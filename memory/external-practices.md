## specmate：稳定结论

### 已由源码或文档证实的机制

- 标准 4 步流程 + `quick` 模式
- Commands / Skills 双入口
- 设计阶段使用 SubAgent 做代码分析与澄清
- P0 / P1 / P2 分级澄清、P2 自动决策记录
- 模板系统 + `specmate validate` 验证引擎
- `planning.md` 批次执行 + 用户“继续”门禁
- archive / specs 归档思路

### 适配当前体系的判断

- **可直接借鉴**：
  - SubAgent 上下文隔离 + 当前 change/group 局部处理
  - `planning.md` 批次执行 + 用户“继续”门禁
- **条件借鉴**：
  - P0 / P1 / P2 分级澄清
  - 4 步精简流程 + Commands / Skills 双入口
  - 模板系统 + `specmate validate` 验证引擎
- **仅保留参考**：
  - `specs/` 作为 Source of Truth 的主仓模式
  - 自动跨会话调用（仓库中未见系统级续接实现）

### 引用时的注意事项

- `specmate` 应拆成两部分看：
  - **执行层流程治理**：4 步流程、澄清分级、上下文治理
  - **知识底座能力**：开发知识库
- 不要把 `specmate` 直接等同于当前 6 阶段闭环。它更适合作为执行层提效参考，不是闭环治理框架的替代品。
- 仓库内存在少量文档/实现不一致，例如：归档命名、知识库输出路径、分组阈值等。引用时优先以“源码已验证机制”表述，不要把实现细节说满。

## 开发知识库：稳定结论

### 已由材料 + 源码对照确认的机制

- 多源输入：前后端代码、PB、SQL、页面访问/Playwright 等
- 11 类文档拆分
- 多 Agent 编排
- 逐文档确认 / To-do List 推进
- 基准文档增量更新 + 只评审 diff 的思路

### 适配当前体系的判断

- **可直接借鉴**：
  - 基准文档增量更新 + 只评审 diff
- **条件借鉴**：
  - 文档拆分 + 多 Agent 编排 + 逐文档确认
  - 复杂存量系统场景下的“知识底座” framing：解决找不到 / 看不懂 / 做不对
- **仅保留参考**：
  - 11 类文档全套一次性引入
  - 所有效率提升、轮次下降、代码可用率提升等量化收益

### 引用时的注意事项

- 当前仓库足以证明“有这套机制设计”，但不能证明用户材料中的所有量化结果。
- 以后在方案或分享中引用这些收益数字时，必须标注为**材料提供数据**，不能写成我们已验证结论。
- 当前最适合吸收的是“拆分知识、按依赖编排、增量更新”三条原则，而不是照搬整套知识库产物结构。
